# 신탁계약서–투자제안서(IM) 정합성 검증을 위한 LLM 하네스 3-Tier 비교 실험 보고서

**프로젝트**: DnB_harness — 한국 사모펀드 신탁계약서 검증 LLM 하네스
**골든셋 버전**: v0.2 (80 케이스)
**실험 일자**: 2026-06-07
**평가 데이터**: `database/gemma4_30reps/` (tier1/tier2/tier3 각 30 reps)

---

## 초록 (Abstract)

사모펀드 설정 과정에서 **신탁계약서와 투자제안서(IM)의 불일치**는 컴플라이언스 사고로 직결되며, 현재는 준법감시인·운용역이 수기로 대조하는 데 문서 1건당 약 3시간이 소요된다. 본 연구는 이 대조 작업을 **LLM 하네스 엔지니어링**으로 자동화하되, "어디까지 LLM에 맡기고 어디까지 결정적(deterministic) 규칙으로 처리할 것인가"라는 아키텍처 질문을 정량적으로 검증한다. 80개 골든 케이스에 대해 세 가지 아키텍처 — (T1) LLM 없는 결정적 캐노니컬 비교, (T2) 결정적 비교 + 선택적 LLM judge, (T3) LLM 전면 정규화 + judge — 를 각 30회 반복 측정하고, 재현율에 2배 가중한 **F2-Score**를 주 지표로, 실무 피로도를 반영한 **업무 효율성 점수(EI)**를 보조 지표로 비교했다.

결과적으로 **T2(결정적+선택 judge)가 F2 0.896 ± 0.008로 최고 성능**을 기록했으며, 이는 T3(0.888 ± 0.005)보다 통계적으로 유의하게 높았고(Welch t = −5.04, p = 7.4×10⁻⁶, Cohen's d = 1.30) EI 또한 더 높았다(96.1% vs 93.1%, p < 10⁻²⁷). 두 LLM 티어 모두 결정적 단독(T1, F2 0.610)을 압도했다(p < 10⁻⁴⁶). 핵심 발견은 **"LLM에 전면 위임(T3)하는 것보다, 결정적 규칙으로 처리 가능한 것은 결정적으로 처리하고 LLM은 표적 폴백으로만 쓰는 것(T2)이 정확도·재현율·실무효율 모두에서 우월하다"**는 점이다.

## 1. 연구 배경 및 리서치 질문

### 1.1 문제 정의

한국 사모펀드는 설정 시 **신탁계약서**(신탁업자·운용사 간 법적 구속 문서)와 **투자제안서(IM)**(투자자 대상 제안 문서)를 별도로 작성한다. 두 문서는 동일한 펀드의 동일한 사실(보수율, 만기, 환매조건, 당사자 등)을 기술해야 하지만, 실무에서는 작성 주체·시점·표기 관행이 달라 **불일치**가 발생한다. 예컨대 계약서의 "연 1,000분의 8.9"(=0.89%)가 IM에 "0.70%"로 잘못 기재되거나, 폐쇄형 펀드인데 IM에 환매조건이 환각(hallucination)으로 신설되는 식이다.

이러한 불일치는 단순 오타가 아니라 **운영 리스크(operational risk)이자 컴플라이언스 사고 요인**이다. 특히 운용지원팀이 없는 소형 운용사에서는 운용역이 직접 두 문서를 수기 대조해야 하며, 본 연구의 측정 기준상 **문서 3종 대조에 약 180분**이 소요된다.

### 1.2 리서치 질문

> **신탁계약서와 IM 간의 불일치는 큰 운영 비용(operation overhead)을 유발한다. 이를 LLM 하네스 엔지니어링으로 해결할 수 있는가? 그리고 그 하네스를 어떤 아키텍처로 구성해야 컴플라이언스 안정성(불일치를 놓치지 않음)과 실무 효율성(헛알람으로 인한 피로 최소화)을 동시에 달성하는가?**

이 질문은 두 개의 하위 질문으로 분해된다.

- **RQ1.** LLM을 결합한 하네스는 결정적 규칙 단독 대비 불일치 검출 성능(재현율·F2)을 유의하게 개선하는가?
- **RQ2.** "결정적 우선 + 선택적 LLM(T2)" 아키텍처와 "LLM 전면 위임(T3)" 아키텍처 중 무엇이 우월한가?

## 2. 평가 지표와 그 의미

### 2.1 혼동행렬 정의 (단일 진실 소스: `docs/GOLDENSET.md §7`)

본 과제에서 **양성(positive)은 "불일치(mismatch)"**다. 즉 우리가 잡아야 하는 사건은 "두 문서가 다르다"이다.

| 기호 | 정의 | 의미 |
|---|---|---|
| TP | gold=mismatch ∩ pred=mismatch | 실제 불일치를 올바로 잡음 |
| FP | gold=match ∩ pred=mismatch | **헛알람** — 같은데 다르다고 함 |
| FN | gold=mismatch ∩ pred≠mismatch | **놓친 불일치** — 컴플라이언스 사고 |
| TN | gold=match ∩ pred=match | 같은 것을 같다고 함 |

> `gold=missing`(정당한 결측)은 재현율 분모·분자에서 모두 제외해 정상 분류에 페널티를 주지 않는다.

### 2.2 각 점수의 의미

- **Precision(정밀도) = TP/(TP+FP)**: 하네스가 "불일치"라고 외친 것 중 진짜 불일치의 비율. 낮으면 헛알람이 많다.
- **Recall(재현율) = TP/(TP+FN)**: 실제 불일치 중 하네스가 잡은 비율. **낮으면 사고를 놓친다.**
- **F1 = 2PR/(P+R)**: 정밀도와 재현율의 조화평균. 둘을 **동등 가중**한다.
- **F2 = 5PR/(4P+R)**: 재현율에 **2배 가중**한 Fβ(β=2). 정밀도보다 재현율을 중시한다.
- **정확도(Accuracy)**: (TP+TN)/전체. 클래스 불균형에 취약.

### 2.3 왜 메인 지표로 F2를 선택했는가 (정확도·재현율보다 나은 이유)

본 과제의 비용 구조는 **비대칭**이다.

- **FN(놓친 불일치)의 비용**은 매우 크다. 잘못된 보수율·만기·환매조건이 그대로 투자자에게 전달되면 **민원·제재·소송**으로 이어진다. 이것이 본 하네스의 존재 이유다.
- **FP(헛알람)의 비용**은 상대적으로 작다. 실무자가 해당 항목을 한 번 더 열어 대조하는 **시간 비용**일 뿐이며(본 연구 기준 건당 2분), 사고로 이어지지 않는다.

이 비대칭성 때문에 지표 선택은 다음과 같이 귀결된다.

- **정확도는 부적합하다.** 골든셋은 정상/변조가 섞여 있고 `missing`이 제외되므로, 정확도는 "많이 맞히면 높아지는" 지표라 **불일치를 놓쳐도(FN) TN이 많으면 높게 나온다.** 즉 사고를 가린다.
- **재현율 단독도 부적합하다.** 재현율만 보면 "모든 것을 불일치라고 외치는" 시스템(T1의 정규화 없는 변형처럼)이 재현율 1.0을 달성하지만 헛알람이 폭증해 실무에서 쓸 수 없다. 재현율은 **정밀도와 함께** 봐야 한다.
- **F1은 정밀도와 재현율을 동등 가중**하므로 본 과제의 비용 비대칭을 반영하지 못한다.
- **F2는 재현율을 2배 가중**하여 "놓치지 않는 것"을 우선하되, 정밀도(헛알람 억제)도 분모에 포함해 균형을 맞춘다. 따라서 **규칙 강화 조치(가드·judge)가 재현율을 끌어올리는 것이 정당한지 입증**하는 데 가장 적합하다.

요약하면, **F2는 "사고를 놓치지 않는 것을 최우선으로 하되, 헛알람 폭증은 막는" 본 과제의 가치 함수를 가장 충실히 반영한 단일 지표**다.

### 2.4 보조 지표: 업무 효율성 점수(EI)

F2가 분류 품질을 보더라도, 실무 도입 가치는 **사람의 시간을 얼마나 아끼는가**로 결정된다. 이를 위해 **헛알람(FP)을 실무자가 재확인하는 시간 비용**을 반영한 EI를 정의한다.

```
EI(%) = (1 − (T_AI + FP × T_Review) / T_Manual) × 100
```

| 상수 | 값 | 의미 |
|---|---|---|
| T_Manual | 180분 | 인간이 문서 3종을 수작업 대조하는 시간 |
| T_AI | 2.5분 | AI 파이프라인 구동·리포트 추출 시간 |
| T_Review | 2분/건 | 가드가 울린 헛알람 1건을 실무자가 재확인하는 시간 |

EI는 "AI 도입 후 실무자가 실제로 쓰는 시간(파이프라인 + 헛알람 재검토)"이 수작업 180분 대비 얼마나 줄었는지를 백분율로 나타낸다. 헛알람이 많을수록 EI는 떨어진다. F2(품질)와 EI(효율)를 함께 보면 **컴플라이언스 안정성과 현업 효율을 동시에** 평가할 수 있다.

## 3. 비교 대상: 3-Tier 하네스 아키텍처

세 티어는 모두 **가드(G1 형식 / G2 출처 / G3 제약)**를 공통 적용하며, **비교 엔진**과 **LLM 사용 범위**에서 갈린다. LLM은 전 티어에서 `claude-sonnet-4-6`을 사용했다.

| | **Tier 1** | **Tier 2** | **Tier 3** |
|---|---|---|---|
| 내부 모드명 | `ontology_policy` | `ontology_policy_judge` | `harness_norm_judge` |
| 비교 엔진 | 결정적 캐노니컬 | 결정적 캐노니컬 | raw 교차검증 |
| 정규화 | 없음(파서 규칙) | 없음(파서 규칙) | **LLM 전 케이스 정규화** |
| LLM judge | **없음** | **review에만 표적 judge** | **남은 review 전부 judge** |
| 철학 | 규칙만 | **결정적 우선 + LLM 폴백** | LLM 중심 |

- **Tier 1 (LLM 없음):** `parse_percent`/`parse_date`/`parse_duration`/`parse_boolean` 등 결정적 파서로 필드 정책(numeric_equal 등)에 따라 비교한다. 수치·날짜는 정확하지만, 이름·유형·엔티티·불리언처럼 의미 판단이 필요한 필드는 `needs_review`로 남아 검출되지 않는다.
- **Tier 2 (결정적 + 선택 judge):** Tier 1과 동일한 결정적 비교를 먼저 수행하고, **결정적으로 풀리지 않은 `needs_review`에 한해서만** 필드 전용 LLM judge(환매가능·만기·환매수수료·범용)를 호출한다.
- **Tier 3 (LLM 전면):** 가드 후 raw 텍스트 교차검증을 하고, **모든 케이스를 LLM으로 정규화**한 뒤 남은 review를 LLM judge로 확정한다. LLM에 정규화 판단을 통째로 위임한다.

## 4. 데이터셋: 골든셋 v0.2 (80 케이스)

### 4.1 구성

골든셋은 채점의 **단일 진실 소스**다. 실제 펀드(이지스블랙ON 1호)의 신탁계약서·IM에서 수기 검증된 14개 필드를 기반으로 한다.

| 구분 | 케이스 수 |
|---|--:|
| 전체 | 80 |
| 정상(category=정상) | 35 (match 31 + missing 4) |
| 변조(category=변조) | 45 (mismatch) |
| 난이도 easy/medium/hard | 11 / 31 / 38 |

14개 필드(펀드명·유형·설정일·만기일·운용사·신탁업자·판매사·운용보수·신탁보수·판매보수·환매가능여부·락업기간·환매주기·환매수수료)를 정상/변조 시나리오로 커버하며, 가드 전용 케이스(G2 가짜인용 5건, G3/SHACL 범위위반 3건)를 포함한다.

### 4.2 무할루시네이션 검증 (v0.1 → v0.2)

v0.1(30 케이스)에서 v0.2(80 케이스)로 확장하며 다음 4중 검증을 적용했다(`scripts/expand_golden.py`).

1. **출처 대조**: 모든 신규 케이스의 `contract_raw`는 PM 확정본에 이미 존재하는 문자열만 재사용(계약서 원문 창작 금지). 계약서 PDF는 스캔본이라 페이지를 이미지로 렌더링해 직접 검증.
2. **실제 하네스 비교**: 수치·날짜·불리언·부재 케이스를 프로젝트의 실제 `compare_values`로 재계산해 gold_label 정합성 확인(이 과정에서 후행 0 "0.890%" 같은 오검출 후보를 잡아 교체).
3. **파서 실측**: 사용한 모든 표기를 실제 파서로 사전 확인(파서가 못 다루는 표기는 배제).
4. **로더·테스트·하네스 정합성**: 80케이스 로더 통과, 전체 테스트 통과, ontology_policy/guard 모드 동작 확인.

## 5. 실험 방법

- **반복**: 각 티어 30회(reps) 측정. 산출물은 `database/gemma4_30reps/tier{1,2,3}/rep_{01..30}.json`(rep별 전체 score.json).
- **LLM 샘플링 온도**: Tier 2·3은 `temperature = 0.7`로 샘플링하여 30개 분포를 생성. **Tier 1은 LLM이 없어 어떤 설정에서도 상수**(분산 0)이며, 고정 결정적 기준선으로 사용.
- **실행**: rep 단위 병렬(5 동시). 모델 `claude-sonnet-4-6`.
- **통계**:
  - 티어별 평균 ± 표본표준편차, 95% 신뢰구간(t 분포).
  - **Tier 3 vs Tier 2**: Welch 2표본 t검정 + Mann–Whitney U(비모수) + Cohen's d(효과크기).
  - **Tier 2/3 vs Tier 1**: Tier 1이 상수이므로 **1표본 t검정**(popmean = Tier 1 값).
- **재현**: `scripts/run_30reps.py`(실행) → `scripts/analyze_30reps.py`(분석, `stats.json` 산출).

## 7. 논의

### 7.1 RQ1 — LLM 결합은 결정적 단독을 압도한다

Tier 1(결정적 단독)은 정밀도 1.000으로 헛알람이 전혀 없지만, **재현율이 0.556에 불과**하다. 즉 실제 불일치 45건 중 20건(44%)을 놓친다. 이는 이름·유형·엔티티·불리언처럼 **의미 판단이 필요한 필드가 결정적 파서로는 `needs_review`에 머물러 검출되지 않기** 때문이다. 컴플라이언스 관점에서 불일치의 44%를 놓치는 시스템은 도입할 수 없다.

LLM judge를 결합한 Tier 2·3은 재현율을 0.885–0.887로 끌어올려 **F2를 약 0.61 → 0.89로 +0.28 개선**했으며, 이는 통계적으로 압도적으로 유의하다(p < 10⁻⁴⁶). **따라서 RQ1에 대한 답은 "그렇다" — LLM 하네스 결합은 결정적 규칙 단독으로는 불가능한 의미 수준 불일치 검출을 가능케 한다.**

### 7.2 RQ2 — "결정적 우선 + 선택적 LLM"(T2)이 "LLM 전면"(T3)보다 우월하다

본 실험의 핵심 발견이다. **Tier 2가 Tier 3보다 F2(0.896 vs 0.888, p = 7.4×10⁻⁶, d = 1.30)와 EI(96.1% vs 93.1%, p < 10⁻²⁷, d = 5.22) 모두에서 유의하게 우월**했다.

원인은 혼동행렬에 있다. **재현율은 두 티어가 사실상 동일**하다(T2 0.885 vs T3 0.887, 차이 0.002는 잡음 수준). 그러나 **헛알람(FP)은 T3가 5.0으로 T2의 2.27보다 두 배 이상 많다.** 즉 Tier 3의 **전면 LLM 정규화가 재현율을 추가로 얻지 못하면서, 정상 케이스를 가끔 오검출(정밀도 0.889 vs 0.946 하락)** 시킨다. 결정적 파서가 정확히 처리할 수 있는 수치·날짜 정규화까지 LLM에 맡긴 대가로 **불필요한 변동성과 헛알람**을 떠안은 것이다.

이는 하네스 엔지니어링의 일반 원칙을 정량적으로 입증한다: **결정적으로 풀 수 있는 것은 결정적으로 풀고(정확·무료·재현가능), LLM은 결정적 방법이 실패하는 의미 판단에만 표적 투입하라.** Tier 2는 이 원칙의 구현이며, 그 결과 정확도·재현율·실무효율을 동시에 최적화한다.

### 7.3 종합 권고

- **Tier 1**은 헛알람 0(EI 98.6%)이지만 재현율이 낮아 **단독 사용 불가**. 다만 1차 고속 스크리닝/회귀 기준선으로 유용.
- **Tier 3**은 재현율은 충분하나 헛알람·변동성이 커 **비용 대비 효율이 낮다.**
- **Tier 2(권고)**: 결정적 정밀도(수치·날짜)와 표적 LLM judge(의미 필드)를 결합해 **F2 0.896, EI 96.1%** 달성. 컴플라이언스 안정성(재현율 0.885)과 현업 효율(헛알람 평균 2.3건, 재검토 ≈ 7분)을 동시 충족하는 **최적 아키텍처**.

EI를 시간으로 환산하면, 수작업 180분 대비 Tier 2 도입 시 실무자는 약 **7분**(파이프라인 2.5분 + 헛알람 재검토 ≈ 4.5분)만 사용한다 — **약 25배의 시간 단축**.

## 8. 한계 및 향후 과제

- **단일 펀드 그라운드트루스**: 골든셋은 1개 펀드(14필드)에서 파생되었다. 변조 케이스는 다양하나, 다른 운용사·상품 유형으로의 일반화는 추가 펀드 확보가 필요하다.
- **judge 의존 케이스의 변동성**: 엔티티·이름·유형·불리언은 LLM judge에 의존하므로 temperature·모델 변경에 민감하다. 본 실험은 temp=0.7에서의 분포를 보고한다.
- **일방 부재 라벨 논의 케이스**(판매사·락업·환매주기 신설, 3건): 계약서에 정당히 부재한 항목을 IM이 신설한 경우로, 현 하네스는 `missing_evidence`로 처리해 자동 검출되지 않는다. gold=mismatch 라벨은 PM 최종 확정 대기 상태이며, 이는 현 하네스의 알려진 사각지대다.
- **κ 합의 미완료**: 골든셋 v0.2는 라벨러 2인 Cohen's κ 합의·freeze 절차(`docs/GOLDENSET.md §6`) 이전 상태다.

## 9. 결론

신탁계약서–IM 불일치라는 운영 리스크는 **LLM 하네스로 해결 가능**하며(RQ1), 그 최적 아키텍처는 LLM 전면 위임이 아니라 **"결정적 우선 + 선택적 LLM judge"**(Tier 2)다(RQ2). Tier 2는 재현율 2배 가중 지표 **F2 0.896**으로 컴플라이언스 안정성을, **EI 96.1%**(수작업 대비 약 25배 단축)로 현업 효율성을 동시에 달성했으며, 이 우월성은 30회 반복 실험에서 통계적으로 유의하다(F2: p = 7.4×10⁻⁶, EI: p < 10⁻²⁷). **F2를 주 지표로 채택한 것은 "불일치를 놓치는 비용 ≫ 헛알람 비용"이라는 본 과제의 비대칭 가치 함수를 정확도·재현율 단독보다 충실히 반영하기 때문**이며, 본 실험은 이 지표 위에서 규칙 강화(가드)와 분류기/judge 결합의 타당성을 데이터로 입증한다.

## 부록 C. 지표 공식

```
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1        = 2·P·R / (P + R)
F2        = 5·P·R / (4·P + R)              # 재현율 2배 가중 (메인 지표)
EI(%)     = (1 − (T_AI + FP·T_Review) / T_Manual) × 100
            T_Manual=180분, T_AI=2.5분, T_Review=2분/건
```
